# 故障排查手册

## 适用范围

本文件记录在重建和发布麻将站点过程中反复出现的问题，以及已经验证有效的处理办法。

## 构建时 `docs/` 下突然出现大量删除

现象：

- `git status` 突然显示 `docs/` 下大量文件被删除

原因：

- `scripts/build_site.sh` 调用 MkDocs 构建时，会先清空站点输出目录再重建

处理办法：

1. 等待构建完全结束
2. 重新执行 `git status`
3. 只以构建完成后的最终状态作为判断依据

## 推送后线上站点看起来还是旧样式

现象：

- 本地 `docs/...` 正确
- 直接抓取线上 HTML/CSS 也已经正确
- 浏览器里却仍然显示旧布局

原因：

- 浏览器缓存了旧版静态资源，尤其是 `stylesheets/extra.css`

处理办法：

1. 对页面进行强制刷新
2. 直接抓线上 HTML 与 CSS，确认实际部署内容
3. 把缓存问题和真实部署失败区分开来

## 牌河表格显示出错，出现字面量 `![...]`

现象：

- 舍牌例题表格里出现原始 Markdown 图片语法
- 或者整张表被渲染成笨重的全宽普通表格

原因：

- 在原始 HTML 表格单元格中写 Markdown 图片语法，未按预期转成 `<img>`
- Material 默认表格样式会把紧凑牌河表格拉宽

处理办法：

1. 在 HTML 表格单元格中使用真实的 `<img>` 标签
2. 外层包裹 `.river-block`
3. 使用 `site_src/docs/stylesheets/extra.css` 中的 `.river-table` 专用样式
4. 检查生成 HTML，确认单元格里确实是 `<img>`

## 首页章节卡片在桌面端和移动端表现不一致

现象：

- 桌面端和移动端卡片布局意外分叉
- 一端变成“左图右文”，另一端仍是纵向卡片

原因：

- 响应式 CSS 修改了基础布局方向，而不是只在目标断点内覆盖

处理办法：

1. 先定义清楚桌面端的基准布局
2. 只有在确实需要时，才在移动端 media query 内改布局方向
3. 每次修改后都验证桌面端与移动端

当前决定：

- 首页教程章节卡片在桌面端和移动端统一使用“图在上、文在下”的纵向布局

## 资源页卡片正文无法横向对齐

现象：

- 资源卡片中的图片高度不一致
- 各卡片正文起点高低不齐

原因：

- 图片宽高比不同
- 只设置 `min-height` 不够，图片仍可能把图片区继续撑高

处理办法：

1. 给图片区使用固定高度，而不是只设 `min-height`
2. 图片使用 `max-width: 100%` 与 `max-height: 100%`
3. 使用 `object-fit: contain` 并居中显示

## 资源缩略图模糊、不完整或依赖外链

现象：

- 原始推荐缩略图分辨率低
- 图片裁切不完整
- 或者页面仍依赖外站图片热链

原因：

- 原始缩略图本身尺寸较小
- 某些旧外链地址已经不稳定

处理办法：

1. 优先使用更高质量的本地替换图
2. 将最终采用的图片镜像到 `raw_site/assets/images/`
3. 页面中通过 `../images/...` 引用
4. 如果临时用了 `imgs/` 之类的中转目录，构建前必须改成规范文件名并挪到正式素材目录

推荐的规范命名：

- `resource-book-*.jpg`
- `resource-game-*.jpg`

## `site_src/docs/images` 行为和预期不一致

现象：

- 修改 `site_src/docs/images` 下的图片时，`raw_site/assets/images` 也跟着变化

原因：

- `site_src/docs/images` 实际上是指向 `../../raw_site/assets/images` 的符号链接

处理办法：

- 把这两个路径视为同一个素材库
- 除非有特殊需求，不要重复保存两份图片

## `docs/sitemap.xml` 与 `docs/sitemap.xml.gz` 总是变化

现象：

- 每次重建后，这两个文件经常显示为已修改

原因：

- 它们是生成产物，即使是很小的改动也可能导致变化

处理办法：

- 在默认工作流中不要提交它们，除非用户明确要求

## 批量补页面级 `description` 后，担心正文被改坏

现象：

- 需要给大量正文页补 SEO 描述
- 但又不希望正文可见内容、排版或翻译文本被改动

原因：

- 页面级 `description` 正确做法是写在 Markdown 头部元数据里
- 如果直接手改正文首段，容易把可见内容也一起改掉

处理办法：

1. 优先使用 `scripts/add_page_descriptions.py`
2. 只给页面新增或更新 YAML 头部中的 `description`
3. 构建后抽样检查生成 HTML 的 `<meta name="description">`
4. 同时确认正文首屏内容没有变化

## `IndexNow` 提交范围不对，把错误页也提交了

现象：

- `IndexNow` 提交脚本把 `404.html` 或其他不应收录的页面也带进去了

原因：

- 站点地图里可能包含技术上可访问、但不应主动推送给搜索引擎的页面

处理办法：

1. 使用 `scripts/submit_indexnow.py` 当前版本
2. 确认提交列表里不包含 `404.html`
3. 如果后续新增 `noindex` 页面，也要同步把过滤规则加进提交脚本

当前约定：

- `IndexNow` 密钥文件位于 `site_src/docs/6f0d3cf671bf4bb3b4dfe2dfef4f11d6.txt`
- 当前站点部署在子路径 `/mahjong_beginner/` 下，因此必须通过 `keyLocation` 指向该子路径中的密钥文件
