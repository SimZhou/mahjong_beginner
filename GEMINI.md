# 麻将入门文档型网站（翻译自 http://beginners.biz/ ）

该目录下是一个可部署的静态页面型日本麻将入门教学网站中文版。网站的内容完全翻译自http://beginners.biz/。

原始网站上包含诸多图片内容，分为以下2种：
1. 麻将章节介绍页面/链接/开头的图片（包含一些美女）
2. 麻将正文介绍页，针对麻将情况的介绍的示意图

针对情况1，忽略原始图片，然后清晰简介的排版，用文字进行替换

针对情况2，需要完全保留原始的图片到翻译后内容的正文中，以保证教程内容的清晰和与原始的一致性。

整体页面的风格和排版要便于阅读，以简约风，文档风为主。可以如同原网站一样，进行不同的章节划分。

智能体在工作时，需要不断迭代循环，检查以下内容：
1. 针对原网页的内容，不能有遗漏，原网页的sitemap见：http://beginners.biz/sitemap/
2. 翻译质量可靠，不能有翻译错误
3. 排版清晰
4. 网页可部署

在完成上述4个目标之前，不断进行和检查自己的工作内容，直到完成为止。

如果有碰到任何问题，请更新该文件AGENTS.md，以便防止后续再犯。

在 push 改动之前，请将改动的页面内容编译到 `docs/` 目录下。

执行git commit和git push来更新页面。

## 本项目已确认的构建与验收规则

1. `mkdocs` 构建配置必须保持 `use_directory_urls: false`
原因：正文里大量图片使用的是 `../hai/...` 和 `../images/...` 这类相对路径。
如果输出成 `.../index.html` 的目录式 URL，这些相对路径会在生成后的页面里指向错误目录，导致后半段页面出现大量示意图丢失。

2. 构建命令统一使用：
`cd _source && /home/zhou/.conda/envs/mahjang/bin/python -m mkdocs build -d ../docs`

3. 每轮大改后都必须做“构建后检查”，不能只检查 `_source/docs/*.md`
至少检查这三项：
- 全站日文残留是否清零或继续下降
- `docs/` 里的正文图片是否存在断链
- `docs/` 里的样式资源是否存在断链

4. 正文示意图属于必须保留资产
凡是原站正文中的战术图、牌姿图、点数图，都必须搬运到 `_source/docs/images/`，并在构建后确认已经同步进入 `docs/images/`。
不能只改 Markdown 引用，不补原图文件。

5. 站点验收时，应优先以构建产物 `docs/` 为准
因为 `_source/docs` 中链接看起来正确，并不代表生成后的 HTML 仍然正确。
真正会部署的是 `docs/`，所以路径、样式、图片都要以 `docs/` 为最终检查对象。

6. 提交时避免把无关文件混入站点更新
像 `GEMINI.md`、临时说明文件、与站点无关的本地实验文件，不应混入这类内容修复提交。
提交范围应优先限定在：
- `_source/docs/`
- `_source/mkdocs.yml`
- `docs/`
- 必要时补充本文件 `AGENTS.md`

## 阶段性经验补充

1. 用户反馈“线上页面仍未变化”时，先检查是否只是本地改动未再次发布
这次已经确认过：有可能页面源码已经修完、本地 `docs/` 也重新生成了，但如果没有再次执行 `git commit` 和 `git push`，线上仍然会停留在旧版本。

2. 针对用户新指出的单页问题，必须走完整闭环，不能只做本地修复
标准步骤必须是：
- 修改 `_source/docs/...`
- 重新编译到 `docs/`
- 检查生成后的对应 `docs/...html`
- `git commit`
- `git push`

3. 验证线上问题时，要区分“本地已修复”和“线上已发布”
本地 `docs/` 里的 HTML 已经变化，不代表 `https://simzhou.com/mahjong_beginner/...` 已经更新。
如果问题针对线上地址，就必须确认对应提交已经 push 到远端。

4. 对于明显机翻且排版错乱的页面，不要只修换行
像 `reach/reach01.md`、`teyaku/teyaku02.md`、`teyaku/teyaku09.md` 这类页面，排版问题往往只是表象，根因是正文仍是严重机翻或 Markdown 结构已损坏。
遇到这种情况，直接对照原始日文页整页重写，比局部补丁更可靠。
